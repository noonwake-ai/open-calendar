# 小红书黑客松｜桌面玄学伙伴 PRD（团队对齐版）

> 这份文档不是对外介绍，也不是产品愿景稿。它的作用只有一个：让设计、前端、后端、硬件同学都知道这次黑客松到底要做什么，各自要交什么。

## 1. 这次我们要做的东西，到底是什么？

我们不是要做一个泛 AI 硬件。我们要做的是一个 **"桌面玄学伙伴"**：

- **左边一块屏**，负责：日历 / 运势 / 特殊日 / 待办 / 摇卦结果
- **右边一个角色舱**，负责：角色待机 / 说话 / 摇卦反馈
- **下方一个打印模块**，负责：把摇卦结果 / 签文 / 标签打印出来

用户看见它时，应该立刻理解成：

> 这是一个会陪我、会提醒我、会给我一个行动出口的桌面玄学设备。

## 2. 这次黑客松版本，核心目标是什么？

黑客松版本只做一件事：

**把"万象有灵 App 里已有的能力"，用一个桌面硬件形态重新组织出来，跑通一个完整体验闭环。**

这个闭环是：

1. 用户站到桌面前
2. 左屏看到今天内容
3. 右侧角色是活的
4. 用户可以问一句 / 摇一次
5. 系统返回一个结果
6. 结果可以打印成一张标签

**只要这条链路顺，Demo 就成立。**

## 3. 和万象有灵 App 后端，怎么连调？

### 3.1 基本原则

这个硬件 **不是新起一套内容系统**。黑客松版本尽量复用 **万象有灵 App 现有后端能力**，硬件端只做：

- 展示
- 简化交互
- 角色表现
- 打印输出

### 3.2 后端应该提供什么

硬件端需要的，不是 App 全量接口，而是几类最小能力：

#### A. 用户与角色基础信息
- 当前用户身份
- 用户八字 / 基础命盘信息（如果已有）
- 当前该唤醒哪个十神角色
- 角色基础素材 / 配置

#### B. 每日首页信息
- 今日日期信息
- 每日运势（感情 / 事业 / 健康 / 财富）
- 特殊日提示（如五合四冲等）
- 今日建议 / 今日提醒

#### C. 待办与长期事项
- 当前用户的待办事项
- 长期事项列表
- 可被设备端展示的"今天该推进什么"建议

#### D. 摇卦能力
- 发起摇卦
- 返回卦象 / 解释 / 结论 / 行动建议
- 如有需要，返回可直接打印的简版文案

#### E. 对话能力（如果本次接）
- 语音转文本后，把文本问题发到后端
- 后端返回角色口吻回复
- 回复文本再给本地 TTS 播放

### 3.3 黑客松版建议的接口策略

建议后端不要给设备端开放太多散接口。最适合的是整理成 **4 个聚合接口**：

1. **设备首页接口**
   - 一次返回今日角色、运势、特殊日、待办摘要
2. **摇卦接口**
   - 输入：问题 / 用户上下文
   - 输出：卦象、解读、行动建议、打印摘要
3. **待办接口**
   - 获取待办列表
   - 新增 / 更新一个待办（如果本次接）
4. **对话接口**
   - 输入文本
   - 输出角色回复文本

这样设备端实现成本最低，调试也最清楚。

### 3.4 对后端同学的要求

后端这次最重要不是"做复杂能力"，而是：

- 先给设备一个 **稳定、简洁、可调试** 的接口层
- 尽量让设备端拿到的是 **已经适合展示的数据**
- 不要把设备前端逼成一个数据拼装层

## 4. 设备上，屏幕到底分别要做什么？

### 4.1 左屏：内容与操作主入口

左屏是这次软件工作的主战场。

#### 左屏必须有的页面 / 模块

**1. 首页**

内容包括：
- 日期 / 时间
- 今日十神角色
- 今日运势 4 卡（感情、事业、健康、财富）
- 特殊日
- 今日待办摘要

这个页面的目标不是"大而全"，而是：

> 用户走到桌前，一眼知道今天最该看什么。

**2. 运势详情**

点击首页的运势卡进入。内容包括：
- 运势分项
- 简短解释
- 今日建议

**3. 特殊日详情**

点击特殊日进入。内容包括：
- 今天是什么特殊日
- 为什么重要
- 有什么提醒

**4. 待办页**

内容包括：
- 待办列表
- 长期事项
- 今日推荐推进项

> 黑客松版这里不需要做复杂任务管理，重点是：让用户看到"玄学内容不是看完就完了，而是会落到行动上"。

**5. 摇卦过程页**

内容包括：
- 摇卦动效
- 过程状态反馈
- 与右舱角色联动

**6. 摇卦结果页**

内容包括：
- 卦象
- 简短结论
- 一句行动建议
- 打印按钮

#### 左屏不建议做的东西
- 长文本输入
- 太多多层级页面
- 太复杂的设置中心
- 太像 App 的全量功能搬运

### 4.2 右侧角色舱：情绪价值与状态反馈

右舱不是信息主屏，它的作用更聚焦：
- 提供陪伴感
- 强化"角色真的在回应我"
- 摇卦和对话时给强反馈

#### 右舱本次必须有的 3 个状态

1. **待机态** — 平时轻微动作，不抢戏，但让设备是"活"的
2. **说话态** — 用户讲话 / 系统回复时切换，最低要求：有口型或明确状态变化
3. **摇卦态 / 施法态** — 摇卦时切换，要有比待机态明显更强的反馈

> **视频素材规范（开发侧）**：视频文件统一放在 `src/AllSpirit/android/app/src/main/assets/video/{tengodId}/` 下，按 4 个场景子目录分类：
> - `idle/` — 待机态（循环播放）
> - `wake/` — 实体按键唤醒态（播完回 idle）
> - `interpret/` — 解读/说话态，左屏翻开运势详情时触发（播完回 idle）
> - `casting/` — 摇卦/施法态，左屏点「解读」时触发（播完回 idle）
>
> Vite build 时会自动将这些视频打包进 `dist-pi/assets/`，树莓派端直接从构建产物提供服务。

#### 右舱这次不追求的东西
- 高复杂剧情系统
- 大量角色状态分支
- 完整 3D 互动

## 5. 这次设计师具体要做什么？

### 5.1 工业设计 / 硬件外观
- 整机外观方向图
- 左右结构关系
- 压杆 / 打印口 / 屏幕位置关系
- 角色舱大致视觉方案

### 5.2 屏幕 UI 设计
至少需要出：首页、运势详情页、待办页、摇卦过程页、摇卦结果页、打印前结果摘要样式

### 5.3 角色视觉设计
至少需要明确：待机态、说话态、摇卦态

### 5.4 打印内容样式
需要设计：签文标签版式、打印内容层级、文案风格

> 目标是：打出来以后，用户愿意留着，而不是看一眼就扔。

## 6. 这次开发具体要做什么？

### 6.1 前端 / 设备端开发
- 左屏页面开发
- 页面切换和状态管理
- 调用后端接口
- 摇卦流程联动
- 打印按钮触发
- 角色状态切换联动

### 6.2 后端开发
- 给设备整理聚合接口
- 输出适合设备展示的数据结构
- 摇卦接口结果标准化
- 待办接口适配
- 对话接口适配（若本次接）

### 6.3 硬件 / 系统开发
- 树莓派运行稳定性
- 双屏输出
- 音频输入输出稳定
- 打印机串口链路闭环
- 实体按键 / 压杆接入

## 7. 当前已知状态

### 已经基本通了
- 树莓派运行与远程联机
- 音频输入（麦克风）
- 音频输出（喇叭）
- 打印机已切到虚拟串口模式：`/dev/ttyACM0`

### 现在还卡的点
- 标签打印协议还没闭环
- 官网示例已经能拿到十六进制样例，但设备尚未稳定出纸
- 设备前端与万象有灵后端的接口定义还没正式收口

## 8. 这次最合理的分工方式

| 角色 | 优先交付 |
|------|---------|
| **设计师** | 1. 整机方向图 2. 左屏首页 + 摇卦结果页 3. 角色三态 4. 打印标签版式 |
| **前端** | 1. 首页 2. 待办页 3. 摇卦过程页 4. 摇卦结果页 |
| **后端** | 1. 首页聚合接口 2. 摇卦接口 3. 待办接口 4. 对话接口（若接）|
| **硬件/系统** | 1. 双屏稳定显示 2. 音频稳定 3. 打印稳定 4. 物理交互件接入 |

## 9. 黑客松版最小验收

只看 5 件事：

1. 左屏首页是否清楚
2. 右舱角色是否有待机 / 说话 / 摇卦 3 态
3. 用户是否能完成一次摇卦
4. 结果是否能形成明确结论页
5. 是否能打印出一张结果标签

**只要这 5 件事成立，这个 Demo 就成立。**

## 10. 最后一句给团队

这次不要把事情做散。**重点不是堆功能，而是把"桌面入口 + 角色反馈 + 摇卦结果 + 标签输出"这条线跑通。**

谁做什么，其实就围绕这一条线展开就够了。
