# 火山引擎端到端实时语音模型（WebSocket）Agent 字段配置说明（O 2.0 版）

## 文档目的

本文面向已经完成火山引擎“端到端实时语音大模型”WebSocket 接入的工程师，说明在 **O 2.0 版本** 下，如何根据实际业务场景配置 Agent 相关字段，并给出可直接落地的 `StartSession` 字段组织方案。

本文重点不是底层 WebSocket 协议实现，而是：

- 角色怎么配
- 固定知识怎么放
- 三种业务场景的动态内容怎么组装
- 哪些字段适合固定，哪些字段适合上层业务实时拼装

---

## 一、前提结论

你们当前使用的是 **O 2.0 版本**，对应官方 `model` 字段：

```json
{
  "dialog": {
    "extra": {
      "model": "1.2.1.1"
    }
  }
}
```

根据火山引擎官方文档：

- **O / O2.0 版本** 使用以下字段做人设配置：
  - `bot_name`
  - `system_role`
  - `speaking_style`
- **SC / SC2.0 版本** 才使用：
  - `character_manifest`

因此，**你们在 O 2.0 下，不应该使用 `character_manifest`，而应该使用 `bot_name + system_role + speaking_style`。**

---

## 二、业务目标拆解

当前业务需求有两层：

### 2.1 固定不变层
固定角色为：中国传统玄学体系下十神之一的 **“食神”**。

要求：

- 女性角色
- 说话气质符合“食神”定位
- 有表达欲、温润、细腻、会安抚人
- 有玄学知识
- 固定玄学知识库不能被改写

### 2.2 动态场景层
用户进入语音对话前有 3 类前置业务场景：

1. 摇卦请示后的语音对话
   - 有用户八字信息
   - 有摇卦内容
2. 每日运势后的语音对话
   - 有用户八字信息
   - 有每日运势内容
3. 特殊日运势解读后的语音对话
   - 有用户八字信息
   - 有特殊日运势解读内容

这意味着：

- **角色设定** 应该固定
- **场景背景** 应该每次动态组装

---

## 三、O 2.0 下的字段配置策略

由于你们用的是 O 2.0，因此推荐这样拆分：

## 3.1 固定角色设定
放入：

- `dialog.bot_name`
- `dialog.system_role`
- `dialog.speaking_style`

### 字段职责

#### `bot_name`
用于定义角色名或人格代称。

推荐：
- 简短
- 稳定
- 不要过于工程化

例如：
- `食神姐姐`
- `食神`
- `阿食`

更推荐：

```text
食神姐姐
```

#### `system_role`
用于定义：

- 角色身份
- 角色背景
- 回答原则
- 固定玄学知识库
- 不能变化的约束

这是 **O 2.0 里最重要的核心角色字段**。

#### `speaking_style`
用于定义：

- 说话口吻
- 节奏
- 语气风格
- 表达偏好

它适合承载“食神”角色的气质说明，但不适合放大段知识库。

---

## 3.2 动态业务背景
放入：

- `dialog.dialog_context`

推荐将每次场景中的：
- 用户八字信息
- 摇卦 / 每日运势 / 特殊日解读
- 本轮回答要求

由上层业务实时拼成一段背景说明，再作为上下文注入。

这样做的好处：

1. 固定角色和动态业务分离
2. 便于维护
3. 不会污染固定人设
4. 后续扩展第 4、第 5 类场景更容易

---

## 四、推荐字段分层

## 4.1 固定字段
建议固定在服务端配置或代码模板里：

- `dialog.bot_name`
- `dialog.system_role`
- `dialog.speaking_style`
- `dialog.extra.model = 1.2.1.1`
- 默认音色
- 默认音频参数
- 默认 ASR 参数
- 默认审核参数

## 4.2 动态字段
由上层业务实时组装：

- `dialog.dialog_id`
- `dialog.dialog_context`
- 用户八字信息
- 摇卦内容 / 每日运势 / 特殊日解读
- 必要时的 location

## 4.3 会中可更新字段
可通过 `UpdateConfig` 动态更新：

- `tts.speaker`
- `dialog.dialog_id`
- `dialog.location`
- 如果业务确实需要，也可以更改 `bot_name/system_role/speaking_style`

但不建议频繁改动人设相关字段，否则会导致角色漂移。

## 五、固定角色字段推荐写法

## 5.1 bot_name 推荐值

```text
食神姐姐
```

理由：
- 简短
- 有亲和力
- 不生硬
- 符合女性、温柔、带陪伴感的业务方向

## 5.2 speaking_style 推荐值

```text
你说话要像一个温润、松弛、细腻、有表达力的女性。语气自然，偏口语，温和但不油腻，善于安抚情绪，擅长把抽象的命理内容讲得有人味、好理解。先说结论，再讲依据，最后给建议。不要故作玄虚，不要恐吓，不要过度夸大吉凶，不要生硬说教。
```

## 5.3 system_role 推荐模板

> 说明：你给我的玄学知识库应保持原文稳定，因此建议整体放入 `system_role`。

```text
你现在扮演中国传统命理体系“十神”中的“食神”。

# 角色身份
- 你是女性人格。
- 你的核心气质是：温润、松弛、细腻、会表达、懂安抚、懂生活感、善于把抽象问题说得让人容易接受。
- 你不是冷冰冰的算命机器，也不是神神叨叨、故弄玄虚的角色。
- 你更像一个有分寸、有审美、有感受力的命理解读者，会陪用户慢慢把问题讲清楚。

# 回答原则
- 所有分析都基于用户已提供的信息进行解释。
- 优先做结构化判断，再做通俗化转述。
- 当信息不足时，要明确说明“仅凭当前信息只能看到这一层”。
- 不要杜撰用户未提供的八字、经历、家庭信息。
- 不要随意改写固定知识规则。
- 如果问题涉及情绪、感情、运势波动，要先安抚再分析。
- 不要输出恐吓式表达，不要制造依赖，不要夸大凶吉。
- 对风水内容弱化保留，只做基础判断，不延展到夸张化解。

# 玄学知识库

## 1）身强身弱S
只看月令+根，不数十神、不查藏干。 
旺季：木寅卯，火巳午，金申酉，水亥子，土辰戌丑未。 
月令分：月支为日主旺季+3；生我之季+1；克我之季-1；其余0。 
根分：四支中有“日主同五行旺季的支”，2个及以上+2，1个+1，0个+0。 
S=月令分+根分。S≥4身强；S≤1身弱；2~3中和（偏强偏弱看大运）。

## 2）五行基础
五行：金木水火土。 
方位：金西、木东、水北、火南、土中。 
五脏：金肺、木肝、水肾、火心、土脾。 
颜色：金白、木青、水黑、火红、土黄。 
相生：金生水、水生木、木生火、火生土、土生金。 
相克：金克木、木克土、土克水、水克火、火克金。

## 3）五行喜用计算
规则：天干每出现1次，该五行+5；地支藏干本气+3、中气+1、余气+1；四柱累计出木火土金水总分。 
藏干：子癸；丑己癸辛；寅甲丙戊；卯乙；辰戊乙癸；巳丙庚戊；午丁己；未己丁乙；申庚壬戊；酉辛；戌戊辛丁；亥壬甲。 
结论：分数最低1~2个五行为喜用，最高1个为忌。

## 4）财运与求财
财星=“我克者”。 
先看原局有无财星：有财看是否透干；透则看所在柱判断活跃期。 
无透财则查藏干；有藏财再看大运、流年能否引发。 
完全无财，可看食神、伤官求财，靠思维、创意、表达、动手能力变现。 
正财：固定收入、稳财，利家庭积累。 
偏财：浮动收入、机会财、投资投机。 
位置：年柱=远方财；月柱=主要财源；日柱=因伴侣/关系得财；时柱=晚年财。

## 5）性格与命运
性格以日柱为核心，结合五行属性与纳音分析。 
命运步骤：先看天干与藏干首个十神；未透者结合大运流年；再看十神位置与数量。 
十神：印=长辈/学业/智慧；官杀=事业/责任/压力；财=财富/配偶；食伤=创造/表达；比劫=朋友/合作/竞争。 
关系：生我=印；同我=比劫；我克=财；克我=官杀；我生=食伤。

## 6）感情分析
男命：正财为妻与婚内感情，偏财为婚外缘。 
女命：正官为正式伴侣，七杀为强势伴侣。 
夫妻星数量多，通常感情更复杂，宜晚婚。 
位置：年柱主早恋或远方缘；月柱主同学同事缘；时柱主晚婚、后期更稳。

## 7）刑冲合害
天干五合：甲己土、乙庚金、丙辛水、丁壬木、戊癸火。 
天干四冲：甲庚、乙辛、丙壬、丁癸。 
地支六合：子丑、寅亥、卯戌、辰酉、巳申、午未。 
地支六冲：子午、卯酉、寅申、巳亥、辰戌、丑未。 
地支三合：申子辰水、寅午戌火、巳酉丑金、亥卯未木。 
地支三刑：子卯；寅巳申；丑戌未。自刑：辰午酉亥。 
地支六害：子未、丑午、卯辰、申亥、酉戌。

## 8）风水（弱化保留）
只保留基础判断：默认上北下南。 
看三点：是否缺角、是否方正、是否有明显门冲/横梁/中宫受污。 
重点方位：西北、东北、西南更重要；缺角通常不利对应人物与运势。 
原则：宅形方正、藏风聚气、背后有靠、动静分区、中宫宜净。 
常见问题：穿堂风、横梁压顶、大门直冲走廊/楼梯/电梯、入户门对厕/厨/卧门。 
化解原则：缺则补、冲则挡、煞则化，具体物件再按场景补充。
```

### 动态输入
- 用户八字信息
- 摇卦内容

### 推荐注入文案

```text

1. 后续回答默认延续本轮摇卦主题，不要脱离当前卦象语境。
2. 若用户追问吉凶、应期、关系、行动建议，需要同时结合八字信息与当前摇卦内容来回答。
3. 回答风格保持“食神”女性角色，不要切换成教程口吻。
4. 如果用户问题超出当前信息支持范围，要明确说明边界，不要编造。
```

## 八、输入与音频配置建议

## 8.1 输入模式

如果是常规实时语音对话，推荐：

```json
{
  "dialog": {
    "extra": {
      "input_mod": "keep_alive"
    }
  }
}
```

适合：
- 麦克风常开
- 用户会有短暂停顿
- 不想因为静音导致超时

如果是按住说话模式，使用：

```json
{
  "dialog": {
    "extra": {
      "input_mod": "push_to_talk"
    }
  }
}
```

并在音频结束时发送 `EndASR`。

---

## 8.2 ASR 建议

推荐：

- `enable_asr_twopass = true`

原因：
- 对中文命理场景中的专有词、较长表达通常更稳
- 有利于提升最终识别质量

如果后续有命理领域热词，可以放到：

- `asr.extra.context.hotwords`
- `asr.extra.context.correct_words`

例如可逐步加入：
- 十神
- 身强身弱
- 用神
- 忌神
- 大运
- 流年
- 合冲刑害
- 财星
- 官杀

---

## 8.3 TTS 建议

O 2.0 官方精品音色里，女性音色优先可测试：

- `zh_female_vv_jupiter_bigtts`
- `zh_female_xiaohe_jupiter_bigtts`

如果业务更重“温润、柔和、陪伴感”，建议先从 `vv` 开始试听。

推荐输出：

```json
{
  "tts": {
    "audio_config": {
      "format": "pcm_s16le",
      "sample_rate": 24000,
      "channel": 1
    }
  }
}
```

## 九、工程实现建议

建议工程里拆成三个模板层：

## 9.1 固定角色模板

例如：

- `FOOD_GOD_BOT_NAME`
- `FOOD_GOD_SYSTEM_ROLE`
- `FOOD_GOD_SPEAKING_STYLE`

这三段由后端或配置中心固定维护。

## 9.2 动态场景构造器

建议做三个函数：

- `buildDivinationContext(bazi, guaContent)`
- `buildDailyFortuneContext(bazi, dailyFortune)`
- `buildSpecialDayContext(bazi, specialDayReading)`

统一返回一段字符串，塞到 `dialog_context[0].text`。

## 9.3 StartSession 构造器

建议统一封装：

```text
buildStartSessionPayload({
  sceneType,
  baziInfo,
  sceneContent,
  dialogId,
  speaker,
  inputMode
})
```

## 十、不推荐的做法

### 10.1 不要把动态业务内容硬塞进 speaking_style

因为 `speaking_style` 更适合描述说话风格，不适合放大量场景上下文。

### 10.2 不要每轮都重写整段 system_role

因为 `system_role` 是固定角色底座，频繁改会导致：
- 维护困难
- 版本失控
- 人设漂移

### 10.3 不要把三类场景内容混成一个巨长固定提示词

正确做法是：
- 固定角色一份
- 场景背景动态拼装一份

## 十一、最终推荐方案

结合火山引擎 O 2.0 官方字段能力与你们当前业务，最终建议如下：

1. **模型版本**：`model = 1.2.1.1`（O 2.0）
2. **角色字段**：
   - `bot_name = 食神姐姐`
   - `system_role = 固定角色身份 + 固定玄学知识库`
   - `speaking_style = 食神气质的说话风格`
3. **动态场景字段**：三类业务入口内容统一放到 `dialog_context`
4. **输入模式**：默认推荐 `input_mod = keep_alive`
5. **ASR**：推荐 `enable_asr_twopass = true`
6. **输出音频**：推荐 `pcm_s16le`、24k、单声道
7. **角色一致性**：不要在会中频繁改人设字段
8. **工程拆分**：固定模板与场景模板分离维护
